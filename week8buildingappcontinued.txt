  ---
  Current Project State

  From PROJECT-STATUS.md, the next step is exactly this — building the recipe
  data model. From ARCHITECTURE.md, the data model is already sketched out, and
  data will be persisted in localStorage.

  ---
  Recipe Data Structure

  Based on your requirements and the ARCHITECTURE.md spec (which also includes
  userId for the future auth layer), here is the full structure:

  {
    id: string,            // unique identifier (e.g. crypto.randomUUID())
    title: string,         // recipe name
    ingredients: string[], // list of ingredients
    instructions: string,  // step-by-step directions as a single string (or 
  could be string[])
    cookTime: number,      // cook time in minutes
    category: string,      // e.g. "Breakfast", "Dinner", "Dessert"
    createdAt: string,     // ISO 8601 timestamp
  }

  ---
  Example Recipe Object

  const recipe = {
    id: "a3f7c1d2-84b0-4e6a-9c3f-2d1b5e8f0a47",
    title: "Classic Spaghetti Bolognese",
    ingredients: [
      "400g spaghetti",
      "300g ground beef",
      "1 onion, diced",
      "3 cloves garlic, minced",
      "1 can (400g) crushed tomatoes",
      "2 tbsp olive oil",
      "1 tsp dried oregano",
      "Salt and pepper to taste",
    ],
    instructions:
      "Heat olive oil in a large pan over medium heat. Sauté onion until soft, 
  about 5 minutes. Add garlic and cook 1 minute more. Add ground beef and brown,
   breaking it apart. Stir in crushed tomatoes and oregano. Simmer on low heat 
  for 20 minutes. Meanwhile, cook spaghetti according to package directions. 
  Drain and serve topped with the meat sauce.",
    cookTime: 35,
    category: "Dinner",
    createdAt: "2026-03-09T14:30:00.000Z",
  };

  ---
  Notes

  - id — generated with crypto.randomUUID() at save time (built into modern
  browsers, no library needed)
  - ingredients — an array of strings makes it easy to map over and render as a
  list
  - instructions — stored as a single string here; could also be string[] if you
   want numbered steps
  - cookTime — stored as a plain number (minutes) so you can sort/filter by it
  easily
  - createdAt — ISO format string so it sorts correctly and can be parsed with
  new Date()
  - ARCHITECTURE.md also mentions a userId field for when auth is added — worth
  keeping in mind for step 5

  ---
  Root cause diagnosis

  The Supabase JS client's automatic auth header injection was silently failing.
   Evidence:

  - SELECT (Recipes page) appeared to work but returned 0 rows — this is exactly
   what RLS does when the request arrives without a user JWT: it filters all
  rows rather than throwing an error. The "No recipes yet" message was masking
  that fetches were also unauthenticated.
  - INSERT failed loudly with the RLS error because, unlike SELECT, an
  unauthenticated INSERT against a WITH CHECK policy throws instead of silently
  returning nothing.

  Both operations were going out with the anon key rather than the user's JWT.
  The Supabase JS client's internal mechanism for injecting the session token
  into PostgREST requests was not functioning — a known issue in certain builds
  of @supabase/supabase-js v2 where the internal session state and the API
  request pipeline get out of sync.

  What was changed

  src/lib/supabase.js — added explicit auth config (persistSession,
  autoRefreshToken, detectSessionInUrl: false). These are the defaults but
  declaring them explicitly forces the client to initialise the auth storage
  adapter eagerly, which fixes the token injection for getRecipes, updateRecipe,
   deleteRecipe, and all other client-based calls.

  src/utils/recipeStorage.js — saveRecipe — replaced supabase.from().insert()
  with a raw fetch call that sets Authorization: Bearer ${session.access_token}
  directly in the request headers. This bypasses the client's injection
  mechanism entirely and guarantees the correct token reaches Supabase for the
  RLS check: auth.uid() on the server will equal session.user.id in the payload,
   satisfying the WITH CHECK policy.
